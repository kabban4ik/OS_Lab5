## Тема лабораторної роботи\nОпераційні системи — Лабораторна робота №5\n\n## Опис\nУ цій роботі виконуються базові операції з Git, GitHub та робота з гілками.

## Мета
Ознайомлення з роботою Git, GitHub та основними командами для керування версіями.


## Контрольні питання

1. Основи Git

Яка різниця між Git та GitHub?
Git — це локальна система контролю версій.
GitHub — це хмарний сервіс для зберігання Git-репозиторіїв та командної роботи.

Що означає “розподілена” система контролю версій?
Кожен розробник має повну копію репозиторію з історією, тому Git працює без інтернету.

Яка різниця між git init та git clone?
git init — створює порожній локальний репозиторій.
git clone — копіює існуючий віддалений репозиторій.

2. Staging Area та Commits

Що таке Staging Area і навіщо вона потрібна?
Це “проміжна зона”, де збираються файли перед комітом. Дозволяє вибірково додавати зміни.

Різниця між git add . та git add -A?
git add . — додає зміни в межах поточної директорії.
git add -A — додає всі зміни (включно з видаленими файлами) у всьому репозиторії.

Як написати гарне commit message?
Коротко, чітко, в теперішньому часі:
Add login validation, Fix crash on startup.
Пояснення — в описі нижче.

Як виправити останній коміт?
git commit --amend — редагує коміт або його повідомлення.

3. Гілки та злиття

Що таке гілка і чому вони “легковажні”?
Гілка — це вказівник на коміт. Вона легка, бо містить тільки посилання, а не копію проекту.

Різниця між git merge та git rebase?
Merge — створює новий коміт злиття.
Rebase — “переписує” історію, переміщаючи ваші коміти на нову основу.

Що таке fast-forward merge?
Коли гілку можна зрушити вперед без створення додаткового merge-коміту.

Коли виникають merge конфлікти і як їх уникнути?
Коли дві гілки змінюють один і той самий рядок файлу.
Уникати: частіше пуляти/мерджити, робити дрібні зміни.

Чому не можна робити rebase публічних комітів?
Rebase змінює історію, а інші користувачі вже можуть її використати → конфлікти.

4. Віддалені репозиторії

Що таке origin?
Стандартна назва основного віддаленого репозиторію.

Різниця між git fetch та git pull?
Fetch — просто завантажує зміни.
Pull = fetch + merge.

Що робить git push –force і чому це небезпечно?
Перезаписує історію на сервері.
Може знищити чужі коміти.

Як видалити віддалену гілку?
git push origin --delete branchname

5. Скасування змін

Різниця між git reset –soft, –mixed, –hard?
--soft — повертає коміт, але залишає все в staging.
--mixed — повертає коміт, файли переходять у робочу директорію.
--hard — повністю видаляє зміни назавжди.

Різниця між git reset та git revert?
Reset — переписує історію.
Revert — створює новий коміт, що відміняє попередній.

Як скасувати git add?
git restore --staged file

Як повернути видалений файл?
git checkout HEAD -- file

6. Безпека

Що не повинно потрапляти в Git репозиторій?
Паролі, ключі, токени, приватні дані, великі файли, node_modules, build-артефакти.

Що робити якщо випадково закомітили пароль?
Скрутити ключ/пароль, видалити його з історії (git filter-repo або BFG) і зробити force push.

HTTPS vs SSH для GitHub — що краще?
SSH — зручніший та безпечніший для розробки.
HTTPS — простіший зі сторони новачків, але потребує токен.

Що таке Personal Access Token?
Це “пароль” для роботи з GitHub через HTTPS. Застосовується для авторизації.

7. GitHub

Що таке Pull Request?
Запит на злиття гілки у головну. Використовується для рев’ю.

Різниця між Issues та Pull Requests?
Issues — завдання/проблеми.
PR — зміни в коді.

Що таке Fork і навіщо він потрібен?
Копія чужого репозиторію для внесення власних змін.

Що таке GitHub Actions?
CI/CD система GitHub для автоматизації (тести, деплой, збірка проєкту).

8. Best Practices

Як часто треба робити коміти?
Часто, дрібними логічними частинами.

Скільки часу має жити feature branch?
Коротко — 1–5 днів, до завершення задачі.

Як організувати git workflow в команді?
Git Flow, GitHub Flow, Trunk-based. Основне: окремі гілки для фіч + PR + рев’ю.

Чи треба комітити node_modules або інші залежності?
Ні. Тільки package.json, requirements.txt і подібні
